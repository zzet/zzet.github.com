В работе с Git часто возникает ситуация, когда нужно отменить последний коммит. Это может быть опечатка в сообщении, неправильно добавленный файл или просто ошибка. Но важно понимать: в Git термин «отмена коммита» охватывает несколько совершенно разных операций.

В зависимости от того, что вы хотите сделать, есть три основных сценария:

1. **Отменить коммит, но сохранить изменения в staging area** (полностью готовы к следующему коммиту) — `git reset --soft HEAD~1`
2. **Отменить коммит и вернуть изменения в рабочую директорию** (изменения не в staging area) — `git reset --mixed HEAD~1`
3. **Полностью удалить коммит и все его изменения** — `git reset --hard HEAD~1`

Кроме того, если коммит уже запушен в удалённый репозиторий, стратегия кардинально меняется. Нельзя просто взять и удалить историю в продакшене — нужно использовать `git revert`.

В этой статье мы разберём все три режима отмены, покажем как работать с несколькими коммитами, и объясним, что делать, если коммит уже опубликован в репозитории.

## Три режима отмены: --soft, --mixed, --hard

Команда `git reset` имеет три основных режима, которые работают по-разному. Давайте посмотрим на каждый:

### git reset --soft: сохранить всё в staging area

```bash
git reset --soft HEAD~1
```

Этот режим откатывает последний коммит, но все его изменения остаются в staging area (`git add` был выполнен). Это полезно, когда вы просто хотите переделать commit message или добавить что-то ещё перед коммитом.

**Пример:**
```bash
$ git log --oneline
a1b2c3d Fix bug in authentication (HEAD -> main)
f4e5d6c Add new feature

$ git reset --soft HEAD~1

$ git status
On branch main
Changes to be committed:
  modified:   src/auth.js
  new file:   tests/auth.test.js
```

После `git reset --soft` вы находитесь в состоянии, как будто только что выполнили `git add .` но ещё не коммитили.

### git reset --mixed: вернуть изменения в рабочую директорию

```bash
git reset --mixed HEAD~1
```

или просто:

```bash
git reset HEAD~1
```

Это режим по умолчанию. Коммит отменяется, но изменения остаются в рабочей директории в виде *модифицированных* файлов (не в staging area). Это самый частый случай — когда вы хотите пересмотреть файлы перед тем как переделать коммит.

**Пример:**
```bash
$ git reset --mixed HEAD~1

$ git status
On branch main
Changes not staged for commit:
  modified:   src/auth.js
  new file:   tests/auth.test.js
```

Теперь вы можете проверить файлы, отредактировать их, и сделать новый коммит.

### git reset --hard: полностью удалить коммит

```bash
git reset --hard HEAD~1
```

Этот режим полностью откатывает коммит и удаляет все его изменения из рабочей директории. Данные теряются полностью. Используйте с осторожностью!

**Пример:**
```bash
$ git reset --hard HEAD~1

$ git status
On branch main
nothing to commit, working tree clean

$ git log --oneline
f4e5d6c Add new feature (HEAD -> main)
```

Все изменения из коммита `a1b2c3d` безвозвратно удалены.

## Сохранить изменения: самый частый случай

Большинство разработчиков используют этот сценарий: отменить коммит, но сохранить возможность переделать его. Для этого используйте:

```bash
git reset HEAD~1
```

или явно:

```bash
git reset --mixed HEAD~1
```

Давайте посмотрим полный пример рабочего процесса:

```bash
# Вы только что коммитили
$ git log --oneline
d7f8e9a Add user profile page
c4b3a2d Fix login form

# Но заметили опечатку в сообщении или пропустили файл
$ git reset HEAD~1

# Проверяем статус
$ git status
On branch main
Changes not staged for commit:
  modified:   src/profile.js
  new file:   src/profile.css

# Редактируем, добавляем недостающие файлы
$ git add src/

# Переделываем коммит с правильным сообщением
$ git commit -m "Add user profile page with styling"

# Проверяем результат
$ git log --oneline
b1c2d3e Add user profile page with styling (HEAD -> main)
c4b3a2d Fix login form
```

## Отменить несколько коммитов

Иногда нужно откатить не один коммит, а несколько последних. Используйте `HEAD~N` где N — количество коммитов:

```bash
# Отменить последние 3 коммита, сохранив изменения
git reset HEAD~3

# Или явно указать режим
git reset --mixed HEAD~3

# Если нужно полностью удалить 3 коммита
git reset --hard HEAD~3
```

Также можно указать конкретный хеш коммита, к которому хотите откатиться:

```bash
# Откатиться к коммиту abc1234 (всё что после него будет отменено)
git reset HEAD abc1234

# Если знаете хеш и хотите сохранить изменения в staging area
git reset --soft abc1234
```

**Практический пример:**
```bash
$ git log --oneline
1f2e3d4 Update dependencies (HEAD -> main)
5a6b7c8 Refactor database module
9d8e7f6 Add caching layer
c4b3a2d Fix login form

# Отменяем последние 3 коммита
$ git reset HEAD~3

$ git status
On branch main
Changes not staged for commit:
  modified:   package.json
  modified:   src/db.js
  modified:   src/cache.js
  new file:   src/cache.test.js

# Все эти файлы теперь можно переделать и создать новый коммит
```

## Если коммит уже запушен в репозиторий

Здесь всё усложняется. Если вы уже выполнили `git push` и коммит попал в удалённый репозиторий (особенно в `main` или `production`), просто использовать `git reset --hard` нельзя. Это нарушит историю для всех, кто имеет копию репозитория.

**Неправильно:**
```bash
# НЕ ДЕЛАЙТЕ ЭТО если коммит уже в удалённом репо!
git reset --hard HEAD~1
git push --force  # Очень опасно!
```

**Правильно: используйте git revert**

```bash
# Отменить последний коммит, создав новый коммит отмены
git revert HEAD

# Или отменить нескольких коммитов
git revert HEAD~2..HEAD

# Запушить новый коммит отмены
git push
```

Команда `git revert` не переписывает историю — она создаёт новый коммит, который отменяет изменения предыдущего коммита. История остаётся чистой, и все видят что произошло.

**Пример:**
```bash
$ git log --oneline
a1b2c3d Fix bug in payment system (HEAD -> main)
f4e5d6c Add checkout page
d7c8e9f Add shopping cart

# Уже запушили, нужно отменить
$ git revert HEAD

# Откроется редактор сообщения
# Можно оставить сообщение по умолчанию или отредактировать

$ git log --oneline
8x9y0z1 Revert "Fix bug in payment system"
a1b2c3d Fix bug in payment system (HEAD -> main)
f4e5d6c Add checkout page
```

Теперь можете запушить:
```bash
git push
```

## Восстановить случайно удалённый коммит

Если вы случайно выполнили `git reset --hard` и потеряли важный коммит, не паникуйте! Git хранит всю историю операций в **reflog** (reference log).

```bash
# Посмотреть все последние операции в репозитории
git reflog

# Вы увидите что-то вроде:
# a1b2c3d (HEAD -> main) HEAD@{0}: reset: moving to HEAD~1
# f4e5d6c HEAD@{1}: commit: Add checkout page
# d7c8e9f HEAD@{2}: commit: Add shopping cart
# 8x9y0z1 HEAD@{3}: reset: moving to HEAD~1
```

Найдите коммит который хотите восстановить и используйте:

```bash
# Восстановить состояние на коммите f4e5d6c
git reset --hard f4e5d6c

# Или используйте HEAD@{N} синтаксис
git reset --hard HEAD@{1}
```

Подробнее смотрите {{< relref "git-reflog" >}}.

## Таблица сравнения: выбираем правильную команду

| Ситуация | Команда | Результат |
|----------|---------|-----------|
| Отменить коммит, оставить changes in staging | `git reset --soft HEAD~1` | Файлы в `git add`, готовы к новому коммиту |
| Отменить коммит, оставить changes unstaged | `git reset HEAD~1` | Файлы модифицированы но не staged |
| Полностью удалить коммит | `git reset --hard HEAD~1` | Коммит и изменения удалены безвозвратно |
| Отменить 3 последних коммита | `git reset HEAD~3` | Все 3 отменены, changes сохранены |
| Откатиться к конкретному коммиту | `git reset abc1234` | К коммиту abc1234, всё после отменено |
| Коммит уже в удалённом репо | `git revert HEAD` | Новый коммит отмены, история сохранена |
| Отменить несколько запушенных | `git revert HEAD~2..HEAD` | Два новых коммита отмены |
| Восстановить потерянный коммит | `git reflog` → `git reset --hard abc1234` | Коммит восстановлен |

## FAQ: Частые вопросы

**Q: В чём разница между git reset и git revert?**

A: `git reset` переписывает историю (удаляет коммиты). `git revert` создаёт новый коммит отмены и сохраняет историю. Используйте `reset` только для локальных коммитов которые никто не видел. Для опубликованных коммитов всегда используйте `revert`.

**Q: Что означает HEAD~1, HEAD~2?**

A: `HEAD~1` — это один коммит назад (последний коммит). `HEAD~2` — два коммита назад. `HEAD~0` — это сам HEAD. Это также можно написать как `HEAD^` (один назад), `HEAD^^` (два назад) и т.д.

**Q: Можно ли восстановить удалённый коммит?**

A: Да! Git хранит всю историю в reflog. Выполните `git reflog`, найдите коммит и используйте `git reset --hard HASH`. Reflog хранит историю обычно 30 дней.

**Q: Что делать если уже запушил с git push --force?**

A: Свяжитесь с членами команды — им нужно выполнить `git fetch --all` или `git pull` чтобы обновить их репозитории. Лучше использовать `git revert` чтобы избежать этой ситуации.

**Q: Как отменить изменения только одного файла?**

A: Используйте `git restore` или `git checkout`:
```bash
git restore --source=HEAD~1 -- path/to/file.js
# или
git checkout HEAD~1 -- path/to/file.js
```

## Дополнительные советы и практики

### Использование git reset в командной работе

Когда вы работаете в команде, важно помнить что `git reset` переписывает историю и может создать проблемы для других разработчиков. Вот лучшие практики:

**Для неопубликованных коммитов (локально):**
- `git reset` безопасен — никто не пострадает
- Используйте его для корректировки перед push

**Для опубликованных коммитов (уже в origin):**
- Никогда не используйте `git reset --hard` с `git push --force`
- Предпочитайте `git revert` для сохранения истории
- Если уж сделали reset и pushed, уведомите команду немедленно

### Комбинирование reset с другими командами

```bash
# Откатить коммит и сделать новый с правильным сообщением
git reset --soft HEAD~1
git commit -m "Правильное сообщение"

# Откатить и выбрать какие файлы коммитить
git reset --mixed HEAD~1
git add src/auth.js
git commit -m "Fix only auth.js"

# Откатить и полностью пересмотреть файлы перед коммитом
git reset HEAD~1
git diff  # Посмотреть что изменилось
git add .
git commit
```

## Заключение

Отмена коммитов в Git — это одна из самых полезных операций, но её важно выполнять правильно:

- **Для локальных коммитов**: используйте `git reset` (--soft, --mixed или --hard в зависимости от нужд)
- **Для опубликованных коммитов**: всегда используйте `git revert`
- **Если потеряли коммит**: спешите к `git reflog`
- **Помните о режимах**: --soft сохраняет в staging, --mixed оставляет unstaged, --hard удаляет всё
- **В командной работе**: информируйте коллег если меняли опубликованную историю

Понимание этих команд делает вас намного более эффективным разработчиком. Не бойтесь экспериментировать в тестовом репозитории!
