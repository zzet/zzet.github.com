Git — это инструмент, который отслеживает изменения в твоих файлах и позволяет вернуться к любой предыдущей версии. Если ты хоть раз называл файлы `проект_финальный.docx`, `проект_финальный_v2.docx`, `проект_финальный_точно_финальный.docx` — Git решает именно эту проблему, только делает это правильно. Его используют миллионы разработчиков по всему миру, и после прочтения этой статьи ты поймёшь почему.

Git расшифровывается просто — это название, которое Линус Торвальдс дал своей утилите. На британском сленге «git» означает «неприятный тип», и Торвальдс с присущим ему юмором назвал проект в честь себя. Но это лирика — важнее понять, что Git делает и зачем он нужен.

## Git — это система контроля версий

Система контроля версий (СКВ) — это программа, которая ведёт историю изменений файлов: кто, когда и что изменил. Самое близкое объяснение — это история правок в Google Docs или режим «Отслеживание изменений» в Word. Только Git работает с любыми файлами и делает это несравнимо мощнее.

Представь, что ты пишешь программу. Ты добавил новую функцию — всё работает. Потом добавил ещё одну — что-то сломалось. Без Git тебе придётся вручную искать, что именно ты изменил. С Git ты просто смотришь историю и в одну команду возвращаешься к рабочей версии.

Три главных дела, которые делает Git:
- сохраняет историю всех изменений проекта
- позволяет нескольким разработчикам работать над одним кодом одновременно
- даёт возможность вернуться к любой предыдущей версии

## История создания Git

Git появился в 2005 году. До этого разработчики ядра Linux использовали коммерческую систему BitKeeper, но лицензионный конфликт заставил сообщество искать альтернативу. Линус Торвальдс — создатель Linux — за несколько недель написал Git с нуля под свои требования.

Требования были жёсткими: скорость, простота архитектуры, поддержка нелинейной разработки (тысячи параллельных веток), полная децентрализация и возможность работать с огромными проектами вроде ядра Linux.

Git справился со всем этим и быстро стал стандартом индустрии. Сегодня его используют не только разработчики, но и технические писатели, аналитики данных, дизайнеры — все, кто работает с файлами, которые меняются со временем.

## Основные компоненты Git

Чтобы понять как работает Git, нужно знать четыре понятия:

**Репозиторий** — папка твоего проекта, за которой следит Git. Внутри есть скрытая папка `.git`, где хранится вся история изменений.

**Рабочая директория** — это обычные файлы проекта, которые ты видишь и редактируешь.

**Индекс (staging area)** — промежуточная зона. Сюда ты складываешь изменения, которые готов сохранить. Это позволяет выбирать, какие именно изменения войдут в следующий снимок.

**Коммит** — это снимок состояния проекта в конкретный момент времени. Каждый коммит имеет уникальный идентификатор, автора, дату и сообщение с описанием изменений.

Вот как они взаимодействуют:

```
Рабочая директория → git add → Индекс → git commit → Репозиторий (история)
```

Ты редактируешь файлы → выбираешь нужные через `git add` → фиксируешь снимок через `git commit`. Всё просто.

## Как Git отслеживает изменения

Git не сохраняет файлы целиком при каждом изменении — он делает снимки состояния всего проекта. Если файл не изменился, Git просто ставит ссылку на предыдущую версию. Это делает его быстрым и экономным.

Каждый снимок (коммит) вычисляет SHA-1 хеш всего содержимого. Это 40-символьная строка вроде `a3f5b2...`. Благодаря этому Git может мгновенно обнаружить любое изменение и гарантировать целостность данных — историю невозможно тихо испортить.

```bash
# Посмотреть что изменилось
git status

# Добавить файлы в индекс
git add main.py

# Создать снимок с описанием
git commit -m "Добавлена функция авторизации"

# Посмотреть историю снимков
git log --oneline
```

## Распределённая природа Git

В отличие от старых систем вроде CVS или SVN, Git — распределённый. Это значит, что каждый разработчик имеет полную копию всего репозитория, включая всю историю. Не нужен постоянный доступ к серверу.

Ты можешь работать офлайн: делать коммиты, просматривать историю, создавать ветки. А когда появится интернет — синхронизироваться с удалённым репозиторием (например, на GitHub).

Централизованные системы хранят всю историю на одном сервере — если он упал, работа встаёт. В Git у каждого полная копия: даже если сервер сгорел, любой участник команды может восстановить весь проект.

## Базовые команды Git

Вот минимальный набор команд, с которого всё начинается:

```bash
# Инициализировать новый репозиторий
git init

# Посмотреть статус — что изменилось
git status

# Добавить файл в индекс (подготовить к коммиту)
git add файл.txt

# Добавить все изменённые файлы
git add .

# Создать коммит — зафиксировать снимок
git commit -m "Описание изменений"

# Посмотреть историю коммитов
git log

# Посмотреть историю компактно
git log --oneline
```

Это не всё, что умеет Git, но с этого набора реально начать работать. Остальное — ветки, слияния, удалённые репозитории — придёт по мере необходимости.

## Git и GitHub — это разные вещи

Один из самых частых вопросов у новичков: «Git и GitHub — одно и то же?». Нет.

**Git** — это программа (инструмент контроля версий), которая работает на твоём компьютере. Она бесплатная и с открытым исходным кодом.

**GitHub** — это веб-сервис для хранения Git-репозиториев в облаке. Он добавляет поверх Git удобный интерфейс, инструменты для совместной работы, баг-трекер и многое другое.

Аналогия: Git — это сам язык, а GitHub — одна из платформ, где люди им пользуются. Существуют и другие похожие платформы: GitLab, Bitbucket, Gitea.

Ты можешь использовать Git без GitHub — локально, на своём компьютере. И это уже даёт тебе полную историю изменений и возможность откатываться назад.

## Зачем Git нужен, если работаешь один

Даже соло-разработчику Git нужен. Вот реальные сценарии:

- Ты добавил новую функцию, что-то сломалось — одна команда `git revert` и ты снова в рабочем состоянии.
- Хочешь попробовать рискованный рефакторинг — создаёшь ветку, экспериментируешь, и если что-то пошло не так, просто удаляешь ветку.
- Через полгода тебе нужно понять, почему была сделана та или иная правка — читаешь историю коммитов с описаниями.
- Работаешь на нескольких машинах — синхронизируешь через GitHub.

Git — это страховка и организация в одном флаконе.

## Часто задаваемые вопросы

### Git и GitHub — это одно и то же?

Нет. Git — это программа для контроля версий, которая работает локально. GitHub — это облачный сервис для хранения Git-репозиториев. Подробнее об этом читай в статье [Что такое репозиторий]({{< relref "chto-takoe-repozitoriy" >}}).

### Можно ли использовать Git для документов Word?

Технически да, но это неудобно. Git отлично работает с текстовыми файлами и кодом. Для Word-документов он тоже отслеживает изменения, но не может показать их в читабельном виде (Word использует бинарный формат).

### Нужен ли интернет для работы с Git?

Нет. Git работает полностью локально. Интернет нужен только для синхронизации с удалённым сервером (GitHub, GitLab) — для команд `git push` и `git pull`.

### Сколько времени нужно, чтобы выучить Git?

Базовые команды — `git init`, `git add`, `git commit`, `git log` — осваиваются за несколько часов. Для комфортной работы в команде (ветки, слияния, разрешение конфликтов) потребуется несколько дней практики.

### Git только для программистов?

Нет. Git используют технические писатели для документации, аналитики данных для скриптов и датасетов, дизайнеры для версионирования конфигов, менеджеры проектов для инфраструктурного кода. Любая работа с файлами, которые меняются, — кандидат для Git.
