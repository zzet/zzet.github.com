Переключение между ветками — одна из самых частых операций при работе с Git. Вы работаете над новой функцией, получаете срочный баг — нужно переключиться на другую ветку. Или просто хотите посмотреть состояние кода в другой ветке. В этой статье разберём все способы переключения и что происходит при этом с вашими файлами.

## Что происходит при переключении ветки

Когда вы переключаетесь на другую ветку, Git делает несколько вещей:

1. Перемещает `HEAD` на новую ветку
2. Обновляет содержимое рабочей директории до состояния последнего коммита в этой ветке
3. Обновляет индекс (staging area)

Это означает, что файлы в вашей папке реально изменяются. Если в ветке `main` файл `app.js` содержит одно, а в `feature/auth` — другое, при переключении между ними файл будет обновляться.

```bash
# Посмотреть текущую ветку
git branch --show-current
# или
git branch
# * main  ← звёздочка показывает текущую
#   feature/auth
```

## Способ 1: git switch (современный, рекомендуется)

Начиная с Git 2.23 (2019 год) появилась специализированная команда для переключения веток:

```bash
git switch feature/auth
```

Это явная, понятная команда, которая делает ровно то, что написано — переключает ветку. В отличие от `git checkout`, она не перегружена дополнительными функциями.

```bash
# Переключиться на main
git switch main

# Переключиться на develop
git switch develop

# Проверить результат
git branch --show-current
# develop
```

### Быстрое переключение на предыдущую ветку

Специальный синтаксис `-` (дефис) означает «предыдущая ветка»:

```bash
git switch main
git switch feature/auth
git switch -    # Вернуться на main
git switch -    # Снова на feature/auth
```

Это как кнопка «Назад» в браузере, очень удобно.

## Способ 2: git checkout (классический)

Старый способ, работает везде и сейчас:

```bash
git checkout feature/auth
git checkout main
```

`git checkout` — многофункциональная команда: она переключает ветки, восстанавливает файлы, переключается на коммиты. Эта многозначность делает её менее понятной для новичков, поэтому и появился `git switch`.

Оба способа делают одно и то же при переключении веток. Используйте тот, который привычнее.

## Переключение на удалённую ветку

Коллега создал ветку `feature/payment` на GitHub, и вы хотите над ней поработать:

```bash
# Сначала получить информацию об удалённых ветках
git fetch origin

# Переключиться на удалённую ветку (Git автоматически создаст локальную с отслеживанием)
git switch feature/payment
# Branch 'feature/payment' set up to track remote branch 'feature/payment' from 'origin'.
# Switched to a new branch 'feature/payment'
```

Если автоматическое создание не сработало:
```bash
# Явно создать локальную ветку с отслеживанием удалённой
git switch -c feature/payment origin/feature/payment

# Или старый способ
git checkout -b feature/payment origin/feature/payment
```

## Защита от потери изменений

Git не позволит переключиться на другую ветку, если у вас есть незакоммиченные изменения, которые конфликтуют с файлами в целевой ветке:

```bash
git switch feature/auth
# error: Your local changes to the following files would be overwritten by checkout:
#   config.js
# Please commit your changes or stash them before you switch branches.
# Aborting
```

У вас три варианта:

### Вариант 1: закоммитить изменения

```bash
git add .
git commit -m "WIP: сохраняю незаконченную работу"
git switch feature/auth
```

### Вариант 2: сохранить изменения через git stash

```bash
git stash         # Временно сохранить изменения
git switch feature/auth
# ... работаете в другой ветке ...
git switch main   # Возвращаетесь
git stash pop     # Восстанавливаете изменения
```

### Вариант 3: отменить изменения (необратимо!)

```bash
git restore .     # Отменить все изменения в рабочей директории
git switch feature/auth
```

Будьте осторожны: `git restore .` удаляет незакоммиченные изменения навсегда.

## Переключение на конкретный коммит (detached HEAD)

Можно переключиться не на ветку, а на конкретный коммит:

```bash
git checkout a1b2c3d
# Note: switching to 'a1b2c3d'.
# You are in 'detached HEAD' state.
```

В этом режиме («отсоединённый HEAD») вы можете просматривать старый код. Если создадите коммиты здесь, они могут быть потеряны при переключении обратно. Чтобы сохранить работу:

```bash
git switch -c new-branch-name
```

## Просмотр всех веток перед переключением

```bash
# Список локальных веток
git branch

# Все ветки: локальные и удалённые
git branch -a

# С информацией о последнем коммите
git branch -v

# Обновить информацию об удалённых ветках
git fetch --prune
git branch -a
```

## Практические примеры

```bash
# Сценарий: переключение для срочного багфикса

# 1. Сохраняем незаконченную работу
git stash

# 2. Переключаемся на main
git switch main

# 3. Создаём ветку для хотфикса
git switch -c hotfix/critical-bug

# 4. Исправляем и коммитим
git add .
git commit -m "fix: critical payment processing error"

# 5. Возвращаемся на feature ветку
git switch feature/new-dashboard

# 6. Восстанавливаем незаконченную работу
git stash pop

# Продолжаем работу над дашбордом
```

## Часто задаваемые вопросы

**Что произойдёт с моими незакоммиченными изменениями при переключении?** Если изменения не конфликтуют с целевой веткой — они сохранятся и перейдут вместе с вами. Если конфликтуют — Git откажет в переключении и предложит закоммитить или сохранить через stash.

**Почему Git не позволяет переключиться?** Наиболее распространённая причина — незакоммиченные изменения в файлах, которые отличаются между ветками. Используйте `git status` чтобы увидеть проблемные файлы, затем закоммитьте или сохраните их.

**Как переключиться на удалённую ветку?** Выполните `git fetch origin` для получения актуального списка веток, затем `git switch имя-ветки`. Git автоматически создаст локальную ветку с отслеживанием удалённой.

**Какая разница между `git checkout` и `git switch`?** `git switch` — специализированная команда только для переключения веток. `git checkout` делает то же самое, но также умеет восстанавливать файлы и переключаться на коммиты. Для переключения веток рекомендуется `git switch` — синтаксис понятнее.

**Как быстро вернуться на предыдущую ветку?** Используйте `git switch -` — аналог кнопки «Назад». Работает и с `git checkout -`.

## Заключение

Для переключения веток используйте `git switch имя-ветки` — это современный, понятный способ. Если работаете со старым кодом или в команде, где привыкли к `git checkout` — он тоже отлично работает.

Перед переключением проверяйте статус через `git status`. Незакоммиченные изменения либо коммитьте, либо сохраняйте через `git stash`. Это предотвратит неприятные сюрпризы.
