Одна из частых задач при работе с Git репозиторием — это переименование и перемещение файлов. На первый взгляд может показаться, что это простая операция, но если сделать её неправильно, можно потерять важную информацию об истории файла. В этой статье разберёмся, как правильно использовать команду `git mv` для сохранения истории коммитов и избежания типичных ошибок.

## Что такое git mv и зачем она нужна

Команда `git mv` — это встроенная Git команда для переименования или перемещения файлов и папок внутри репозитория. На первый взгляд может показаться, что это просто удобная обёртка над обычной системной командой `mv`, но на самом деле это гораздо больше.

### Сравнение: git mv vs ручное переименование

Давайте посмотрим, чем отличается использование `git mv` от стандартного подхода с `mv`, `git rm` и `git add`:

**Неправильный способ (ручное переименование):**
```bash
mv old_name.txt new_name.txt
git add new_name.txt
git rm old_name.txt
git commit -m "rename file"
```

**Правильный способ (git mv):**
```bash
git mv old_name.txt new_name.txt
git commit -m "rename file"
```

Хотя оба способа приводят к одинаковому результату на рабочем дереве, Git интерпретирует их по-разному. При использовании `git mv` Git автоматически определяет, что файл был переименован, и сохраняет эту информацию. При ручном переименовании Git может воспринять это как удаление старого файла и создание нового, что может привести к потере истории при использовании `git log --follow`.

### Почему git mv лучше сохраняет историю

Git использует алгоритм сравнения содержимого для определения переименования файлов. Когда вы используете `git mv`:

1. Git явно помечает операцию как переименование
2. Команда `git log` может легко отследить историю файла в обе стороны
3. При слиянии веток (merge) Git лучше разрешает конфликты с переименованными файлами
4. История истории совместимости с визуальными инструментами для просмотра истории

## Синтаксис команды git mv

Базовый синтаксис команды очень простой:

```bash
git mv <источник> <назначение>
```

Команда также поддерживает несколько флагов для специальных случаев:

- `-f` или `--force` — заставить перезаписать целевой файл, если он уже существует
- `-v` или `--verbose` — выводить информацию о перемещаемых файлах
- `-n` или `--dry-run` — показать что произойдет, без выполнения команды
- `-k` — пропустить путь, если это приведёт к ошибкам

## Переименование файла через git mv

Самый простой сценарий — переименование файла в том же каталоге. Например, переименуем файл `README.md` в `README.ru.md`:

```bash
git mv README.md README.ru.md
git status
```

После выполнения этих команд вы увидите:

```
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        renamed:    README.md -> README.ru.md
```

Обратите внимание, что Git уже автоматически добавил изменение в staging area. Теперь можно сделать коммит:

```bash
git commit -m "rename README.md to README.ru.md"
```

Важный момент: после переименования файл находится в staging area (как результат `git add`), поэтому вам не нужно делать отдельный `git add` для переименованного файла.

## Перемещение файла в другую папку

Часто требуется не только переименовать файл, но и переместить его в другой каталог. Например, переместим утилиту из корня проекта в папку `lib`:

```bash
git mv src/utils.js lib/utils.js
git status
git commit -m "move utils.js to lib directory"
```

Или переместим компонент React в более правильную структуру папок:

```bash
git mv components/Button.jsx src/components/Button.jsx
git status
git commit -m "move Button component to src/components"
```

Вы можете комбинировать переименование и перемещение в одной команде:

```bash
git mv components/OldButton.jsx src/components/NewButton.jsx
```

Это одновременно переместит файл в новую папку и переименует его.

## Перемещение нескольких файлов сразу

Git mv также может работать с несколькими файлами и глобальными шаблонами. Переместим все Markdown файлы в папку `docs`:

```bash
git mv *.md docs/
```

Или переместим все JavaScript файлы вспомогательных функций:

```bash
git mv src/helpers/*.js lib/helpers/
```

Для каждого перемещенного файла Git создаст отдельную запись о переименовании в staging area.

## Перемещение целой папки

Иногда требуется переместить целую папку со всеми её содержимым. Git mv отлично справляется с этой задачей:

```bash
git mv old-folder/ new-folder/
```

Например, если у вас есть устаревший код, который нужно переместить в архив:

```bash
git mv src/legacy/ archive/legacy/
git status
git commit -m "archive legacy code"
```

Git автоматически обновит информацию обо всех файлах в этой папке и их истории.

## Просмотр истории переименованного файла

После переименования файла может возникнуть вопрос: как посмотреть историю всех коммитов, которые касались этого файла, в том числе когда он ещё имел старое имя? Для этого используется флаг `--follow`:

```bash
git log --follow new_name.txt
```

Вывод будет содержать все коммиты, касающиеся файла, как под новым названием, так и под старым. Без флага `--follow` Git не будет показывать коммиты из времени, когда файл имел другое имя.

Для более компактного вывода используйте флаг `--oneline`:

```bash
git log --follow --oneline new_name.txt
```

Если нужно посмотреть изменения (diff) для каждого коммита, используйте флаг `-p`:

```bash
git log --follow -p new_name.txt
```

Флаг `--follow` необходим потому, что Git по умолчанию отслеживает файлы по пути, а не по их содержимому. Когда файл переименовывается, его путь меняется, и без специального флага Git не свяжет старую и новую версию в истории.

Вы также можете использовать `git show` для просмотра конкретного коммита, где произошло переименование:

```bash
git show HEAD
```

В выводе вы увидите, что файл был переименован:

```
rename from old_name.txt
rename to new_name.txt
```

## Отмена git mv до коммита

Если вы выполнили `git mv`, но потом передумали, есть несколько способов отменить операцию.

**Способ 1: Использование git restore**
```bash
git restore --staged new_name.txt old_name.txt
```

**Способ 2: Просто выполнить обратное перемещение**
```bash
git mv new_name.txt old_name.txt
git reset HEAD
```

**Способ 3: Использование git reset**
```bash
git reset HEAD
git checkout .
```

Последний способ более кардинальный и отменит все изменения в индексе и рабочем каталоге.

Если переименование уже закоммичено, отменить его можно через `git revert` или `git reset`, но это выходит за рамки данной статьи.

## Типичные ошибки при использовании git mv

### fatal: not under version control

Эта ошибка возникает, если вы пытаетесь переместить файл, который не был добавлен в Git:

```bash
git mv newfile.txt moved.txt
# fatal: not under version control, source=newfile.txt
```

Решение: сначала добавьте файл в Git:

```bash
git add newfile.txt
git mv newfile.txt moved.txt
```

### fatal: destination exists

Ошибка возникает, когда целевой файл уже существует:

```bash
git mv old.txt new.txt
# fatal: destination exists, source=old.txt, destination=new.txt
```

Решение: используйте флаг `-f` для переписания:

```bash
git mv -f old.txt new.txt
```

### Конфликты при слиянии веток

Если разные разработчики переименовывают один и тот же файл по-разному в разных ветках, при слиянии может возникнуть конфликт. Git постарается разрешить его автоматически, но иногда может потребоваться ручное вмешательство.

## git mv vs ручное переименование

Давайте создадим сравнительную таблицу:

| Характеристика | git mv | mv + git rm + git add |
|---|---|---|
| Простота | Одна команда | Три команды |
| История файла | Сохраняется как переименование | Может быть потеряна |
| git log --follow | Работает корректно | Требует флага --follow |
| Слияние веток | Лучше разрешает конфликты | Может быть проблем |
| Staging area | Автоматически добавляет | Требует явно |

**Что происходит под капотом:**

Когда вы используете `git mv`, Git:
1. Перемещает файл в файловой системе
2. Обновляет индекс (staging area) с информацией о переименовании
3. Помечает операцию как переименование для последующего коммита

Когда вы используете ручное переименование (`mv` + `git rm` + `git add`), Git:
1. Видит удаление старого файла
2. Видит создание нового файла с похожим содержимым
3. Пытается определить переименование по содержимому (это не всегда срабатывает)

## Практические примеры

**Пример 1: Реорганизация структуры проекта**

```bash
# Переместим все компоненты в папку components
git mv Button.jsx components/Button.jsx
git mv Card.jsx components/Card.jsx
git mv Modal.jsx components/Modal.jsx
git commit -m "move components to components folder"
```

**Пример 2: Переименование с расширением файла**

```bash
# Переименуем JavaScript файл в TypeScript
git mv index.js index.ts
git commit -m "convert index.js to TypeScript"
```

**Пример 3: Множественное перемещение**

```bash
# Переместим все конфигурационные файлы в папку config
git mv .env config/.env.local
git mv .babelrc config/.babelrc
git mv webpack.config.js config/webpack.config.js
git commit -m "move configuration files to config directory"
```

## Часто задаваемые вопросы

**Q: Сохраняет ли git mv историю коммитов?**

A: Да, `git mv` явно помечает операцию как переименование и сохраняет полную историю коммитов. Для просмотра истории переименованного файла используйте `git log --follow новое_имя.txt`.

**Q: Как переименовать файл через git и не потерять историю?**

A: Используйте `git mv старое_имя новое_имя` вместо ручного переименования командой `mv`. Это гарантирует, что Git правильно отследит переименование и сохранит историю.

**Q: Чем git mv отличается от mv + git add?**

A: `git mv` — это одна команда, которая автоматически обновляет индекс и помечает операцию как переименование. При использовании ручного `mv` + `git rm` + `git add` Git может не распознать операцию как переименование, особенно если содержимое файла было изменено.

**Q: Как git mv работает под капотом?**

A: `git mv` выполняет физическое перемещение файла в файловой системе и обновляет Git индекс с информацией о переименовании. Это помечается как операция переименования для последующего коммита, что позволяет Git правильно обработать изменения при слиянии веток и отслеживании истории.

**Q: Как отменить git mv до коммита?**

A: Используйте команду `git mv новое_имя старое_имя` для обратного перемещения, а затем `git reset HEAD` для удаления из staging area. Или используйте `git restore --staged` для удаления из индекса.

**Q: Работает ли git mv с папками?**

A: Да, `git mv` работает с папками так же, как с файлами. Вы можете переместить целую папку со всем её содержимым одной командой: `git mv старая_папка/ новая_папка/`.

**Q: Как посмотреть историю файла после переименования?**

A: Используйте `git log --follow имя_файла`. Флаг `--follow` указывает Git отследить переименования и показать историю файла под всеми его предыдущими именами.

**Q: Можно ли переименовать файл в несколько целей одновременно?**

A: Нет, `git mv` переместит один файл в одно место. Для перемещения нескольких файлов используйте несколько команд `git mv` или глобальные шаблоны: `git mv *.txt docs/`.

**Q: Что делать, если целевой файл уже существует?**

A: Используйте флаг `-f` для переписания: `git mv -f старый новый`. Это перезапишет целевой файл содержимым исходного.

**Q: Как проверить что произойдет без выполнения команды?**

A: Используйте флаг `-n` или `--dry-run`: `git mv -n старый новый`. Это покажет что произойдет без реального выполнения команды.

## Заключение

Команда `git mv` — это простой и эффективный способ переименовывать и перемещать файлы в Git репозитории с сохранением полной истории. Вместо ручного использования системной команды `mv` с последующими `git rm` и `git add`, используйте `git mv` для лучшего управления историей файлов и избежания потенциальных проблем при слиянии веток.

Помните ключевые моменты:
- `git mv` автоматически помечает операцию как переименование
- Используйте `git log --follow` для просмотра полной истории файла
- Флаг `-f` помогает при конфликтах с существующими файлами
- `git mv` работает с файлами, папками и глобальными шаблонами
