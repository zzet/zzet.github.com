Случилось — вы случайно удалили ветку в Git, и теперь паникуете, что потеряли все свои изменения. Глубоко вдохните. В большинстве случаев удалённую ветку **можно полностью восстановить**, даже если это произошло совсем недавно. Git предусмотрел для этого несколько механизмов, и мы разберём их все.

## Можно ли восстановить удалённую ветку в Git?

**Короткий ответ: да, в большинстве случаев можно.**

Вот почему это возможно:

1. **Git не удаляет коммиты сразу**. Когда вы удаляете ветку (делаете `git branch -d feature`), вы удаляете только *указатель* на ветку, а сами коммиты остаются в объектной базе данных Git.

2. **Reflog ведёт историю всех движений HEAD**. Это логирует каждый переход между ветками, каждый reset, каждый rebase. Это невероятно полезный механизм для восстановления.

3. **Есть сроки хранения**. По умолчанию запись в reflog хранится **90 дней** (контролируется переменной `reflog.expire`). Если вы удалили ветку неделю назад — она спокойно восстановится. Если полгода назад — с большей вероятностью нет.

Исключение: если вы выполняли агрессивную сборку мусора (`git gc --aggressive`) или явно удаляли ветку с флагом `-D`, процесс может быть сложнее, но всё ещё возможен.

## Способ 1: Восстановление через git reflog (основной метод)

Это самый надёжный способ для локальных удалений. **Reflog** — это рефлог (reference log), история всех перемещений указателя HEAD.

### Шаг 1: Посмотрим reflog

```bash
git reflog
```

Вы увидите примерно такой вывод:

```
abc1234 HEAD@{0}: checkout: moving from feature-login to main
def5678 HEAD@{1}: commit: Add password validation
ghi9012 HEAD@{2}: checkout: moving from main to feature-login
```

Каждая строка — это событие. Справа видны действия, которые были выполнены.

### Шаг 2: Найдём последний коммит удалённой ветки

Ищите строки, где упоминается ваша ветка. Может выглядеть так:

```
feature-login: moving from feature-login to main
# или
branch: Created from...
# или просто последний коммит в этой ветке
```

Запомните хэш коммита (в примере выше это `abc1234`).

### Шаг 3: Восстанавливаем ветку

Просто создаём новую ветку, указав найденный хэш:

```bash
# Вариант 1: создание и переход в один шаг
git checkout -b feature-login abc1234

# Вариант 2: просто создание ветки
git branch feature-login abc1234

# Вариант 3: modern syntax (Git 2.23+)
git switch -c feature-login abc1234
```

**Готово!** Ваша ветка восстановлена со всеми коммитами.

## Способ 2: Поиск через git log --all

Если reflog не помог или вы хотите найти ветку по содержимому коммитов, используйте `git log --all`.

```bash
# Показать все коммиты в репозитории (включая те, на которые больше не ссылаются ветки)
git log --all --oneline

# Результат:
# f1a2b3c Fix typo in README
# e4d5c6b Add new feature
# d3c4b5a Remove old code
# ...

# Можете отфильтровать по сообщению коммита
git log --all --oneline | grep "password validation"

# Или поиск по автору
git log --all --oneline --author="Иван" | head -20
```

Как только нашли нужный коммит по сообщению или автору, создаёте ветку:

```bash
git checkout -b feature-restored f1a2b3c
```

**Совет:** Используйте также `git log --all --graph` для визуализации дерева коммитов — так легче понять, какие коммиты принадлежали удалённой ветке.

## Способ 3: Восстановление с удалённого сервера

Если вы работали в команде и ветка была запушена на GitHub/GitLab/Bitbucket, она всё ещё там!

```bash
# Синхронизируемся с remote
git fetch origin

# Смотрим, какие ветки есть на сервере
git branch -r
# Увидим: origin/feature-login

# Создаём локальную ветку из remote
git checkout -b feature-login origin/feature-login

# Или modern syntax:
git switch -c feature-login origin/feature-login
```

Это абсолютно безопасный способ — ветка на сервере ничего не знает об удалении локально.

## Способ 4: Восстановление удалённой ветки на GitHub через веб-интерфейс

Если вы удалили ветку **прямо на GitHub** (через веб-интерфейс), GitHub хранит её в течение **30 дней**.

### Восстановление:

1. Откройте репозиторий на GitHub
2. Перейдите в **Branches** (Ветки)
3. Прокрутите вниз до **Deleted branches** (Удалённые ветки)
4. Найдите нужную ветку
5. Нажмите кнопку **Restore** (Восстановить)

**Важно:** Этот способ работает только если:
- Ветка была удалена через интерфейс GitHub
- Прошло меньше 30 дней
- Вы администратор репозитория

Если вы удалили ветку через команду `git push origin :feature-login` (или `git push origin --delete`), GitHub тоже поймёт это как "удаление на сервере" и покажет в "Deleted branches".

## Способ 5: git fsck для поиска потерянных объектов

Если ни один из предыдущих способов не помог (reflog истёк, ветка была удалена давно), есть последняя линия защиты.

```bash
# Найти все "потерянные" объекты в базе
git fsck --lost-found

# Результат:
# dangling commit abc1234567890
# dangling blob def5678901234
# ...
```

`Dangling commit` — это коммит, на который не ссылается никакая ветка, метка или рефлог. Это может быть ваша удалённая ветка!

```bash
# Посмотрим подробнее на каждый потерянный коммит
git fsck --lost-found | grep "commit"

# Для каждого коммита можно посмотреть содержимое
git log -1 abc1234567890

# И если это нужный коммит, создаём ветку
git branch recovered-feature abc1234567890
```

Это долгий процесс, но он работает даже когда reflog истёк.

## Восстановление ветки: шаг за шагом для паники

Если вы сейчас в стрессовом состоянии и хотите быстро восстановить ветку:

**Шаг 1:** Выполните `git reflog` и скопируйте хэш последнего коммита в нужной ветке.

**Шаг 2:** Выполните `git checkout -b имя-ветки хэш`.

**Готово.** 99% случаев решаются за 30 секунд.

Если это не сработало:

**Шаг 3:** Выполните `git fetch origin` и посмотрите на `git branch -r` — может быть, ветка всё ещё на сервере?

**Шаг 4:** Если и это не помогло, идите на GitHub и проверьте вкладку "Deleted branches".

Всё по-прежнему не помогло? Тогда `git fsck --lost-found` — ваш последний шанс.

## Профилактика: как не удалять ветки случайно

Конечно же, лучше всего не допускать удалений. Вот несколько простых правил:

### 1. Помните различие между `-d` и `-D`

```bash
# -d: безопасное удаление (удалит только если ветка слита в main/develop)
git branch -d feature-login

# -D: принудительное удаление (удалит даже если есть неслитые коммиты!)
git branch -D feature-login
```

**Совет:** Всегда используйте `-d` по умолчанию. Это спасёт вас от случайного удаления важных веток.

### 2. Проверьте, слита ли ветка перед удалением

```bash
# Показать все слитые в текущую ветку
git branch --merged

# Показать все ещё не слитые ветки
git branch --no-merged
```

Это поможет вам понять, какие ветки уже включены в main, а какие содержат уникальные изменения.

### 3. Пушьте в remote перед удалением локально

```bash
# Перед удалением локально
git push origin feature-login

# Только потом удаляйте локально
git branch -d feature-login
```

Если ветка на сервере, вы всегда сможете её восстановить, как мы показали в Способе 3.

### 4. Используйте git aliases для безопасного удаления

```bash
# Добавьте в .gitconfig:
git config --global alias.safe-delete '!git branch --no-merged | grep -v "^*" | fzf -m | xargs git branch -d'
```

Это создаст команду `git safe-delete`, которая покажет вам все неслитые ветки в интерактивном меню.

## FAQ: Частые вопросы

### 1. Как восстановить удалённую ветку git?

Используйте `git reflog`, найдите хэш последнего коммита, а затем создайте ветку: `git checkout -b имя-ветки хэш`. Это работает в 99% случаев.

Если reflog не помог, проверьте на remote: `git fetch origin && git checkout -b имя-ветки origin/имя-ветки`.

Полная инструкция выше в {{< relref "vosstanovit-vetku-git" >}} — прочитайте все способы.

### 2. Сколько времени хранятся удалённые ветки в reflog?

**По умолчанию: 90 дней.** Это контролируется переменной Git `reflog.expire`:

```bash
# Посмотреть текущее значение
git config core.logallrefupdates

# Изменить на 180 дней (более долгое хранение)
git config --global gc.reflogExpire 180.days
```

### 3. Что делать если удалённой ветки нет в reflog?

Используйте `git log --all --oneline` для поиска по содержимому коммитов. Если и это не помогло, используйте `git fsck --lost-found` для поиска потерянных объектов.

В крайнем случае, если ветка была запушена, она должна быть на сервере (GitHub, GitLab и т.д.).

### 4. Как восстановить удалённую ветку на GitHub?

Есть два способа:

**А) Через веб-интерфейс GitHub:**
- Перейдите в Branches
- Найдите раздел "Deleted branches"
- Нажмите "Restore"

**Б) Через командную строку:**
```bash
git fetch origin
git checkout -b branch-name origin/branch-name
```

GitHub хранит удалённые ветки 30 дней.

### 5. Можно ли восстановить ветку через git stash?

**Нет, это разные вещи.** `git stash` хранит незакоммиченные изменения, а не ветки. Но если у вас были незакоммиченные изменения на удалённой ветке, они могут быть в `git stash`:

```bash
git stash list          # Посмотреть все stashes
git stash show stash@{0}  # Посмотреть содержимое
git stash pop stash@{0}   # Применить и удалить
```

Это не восстановит ветку, но может спасти ваши изменения.

### 6. Как избежать случайного удаления веток в команде?

**Лучшие практики:**

1. **Используйте protected branches** на GitHub/GitLab — запретите удаление важных веток (main, develop).

2. **Используйте `-d` вместо `-D`** — это заставит Git проверить, слита ли ветка.

3. **Требуйте pull requests** — не позволяйте пушить напрямую в main.

4. **Настройте branch protection rules:**
   - На GitHub: Settings → Branches → Branch protection rules
   - Требуйте одобрение перед merge
   - Требуйте, чтобы статус checks были зелёные

5. **Используйте pre-push hooks** — локальный скрипт, который предупредит перед удалением:

```bash
# .git/hooks/pre-push
#!/bin/bash
if [[ $1 == "origin" ]]; then
  if [[ $2 == *"--delete"* ]] || [[ $3 == *":"* ]]; then
    echo "⚠️ Вы пытаетесь удалить ветку!"
    echo "Уверены? (y/n)"
    read -r response
    if [[ ! $response =~ ^[Yy]$ ]]; then
      exit 1
    fi
  fi
fi
```

### 7. Что если я удалил ветку через `git push origin :branch`?

Это удаление на удалённом сервере, но локально ветка остаётся! Восстановление:

```bash
git branch                    # Ветка всё ещё здесь локально
git push origin branch-name   # Запушьте её обратно на сервер
```

Или если вы уже удалили и локально, используйте reflog или GitHub's "Deleted branches".

## Интеграция с рабочим процессом

После восстановления ветки убедитесь, что:

1. **Ветка актуальна:**
   ```bash
   git checkout feature-restored
   git pull origin main  # Или ваша base branch
   ```

2. **История коммитов чистая:**
   ```bash
   git log --oneline -10  # Проверьте последние коммиты
   ```

3. **Нет конфликтов:**
   ```bash
   git status  # Должно быть "clean"
   ```

4. **Запушьте в remote:**
   ```bash
   git push origin feature-restored
   ```

Теперь ваша ветка полностью восстановлена и безопасна.

## Заключение

Удаление ветки в Git — это не конец света. В большинстве случаев (если прошло не слишком много времени) ветка восстанавливается буквально за несколько команд. Главное — не паниковать и помнить про reflog.

**Запомните:**
- **reflog** — ваш лучший друг для восстановления локальных удалений
- **remote** — ваша страховка, если ветка была запушена
- **GitHub's UI** — самый простой способ для восстановления на сервере
- **git fsck** — последняя линия защиты для очень старых удалений

И в следующий раз используйте профилактику: пушьте в remote, используйте `-d` вместо `-D`, и настройте protected branches.

Ваша работа в безопасности.
