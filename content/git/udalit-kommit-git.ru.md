Случайно закоммитили секретный ключ? Или добавили несколько черновых коммитов, которые нужно убрать? Git предоставляет несколько инструментов для удаления коммитов. Выбор метода зависит от одного ключевого вопроса: ветка уже опубликована или нет?

## Главное правило перед удалением коммитов

**Для локальных (неопубликованных) коммитов** — используйте `git reset`. Это быстро и безопасно, так как никто ещё не видел эти коммиты.

**Для опубликованных коммитов** (ветка уже в origin, другие видели коммиты) — используйте `git revert`. Он создаёт новый коммит, отменяющий изменения, не переписывая историю.

```bash
# Перед удалением — всегда проверить что будет затронуто
git log --oneline -5
git status
```

## Удаление последнего локального коммита

`git reset` перемещает указатель HEAD назад в истории. Есть три режима работы:

**`--soft`** — коммит удалён, изменения остаются в индексе:
```bash
# Удалить последний коммит, сохранить изменения в индексе
git reset --soft HEAD~1

# Теперь изменения готовы для нового коммита
git status  # Changes to be committed...
git commit -m "Правильное сообщение коммита"
```

**`--mixed`** (режим по умолчанию) — коммит удалён, изменения в рабочей директории, но не в индексе:
```bash
# Удалить последний коммит, сохранить изменения в файлах
git reset --mixed HEAD~1
# или
git reset HEAD~1

# Теперь изменения в файлах, но не staged
git status  # Changes not staged for commit...
```

**`--hard`** — коммит удалён, все изменения из него тоже удалены:
```bash
# Удалить последний коммит ВМЕСТЕ с изменениями (необратимо!)
git reset --hard HEAD~1

# Убедитесь, что эти изменения вам точно не нужны
```

## Удаление нескольких последних коммитов

Число после `HEAD~` указывает, сколько коммитов удалить:

```bash
# Удалить последние 2 коммита
git reset --hard HEAD~2

# Удалить последние 3 коммита, сохранив изменения
git reset --soft HEAD~3

# Удалить до конкретного коммита по хешу
git reset --hard a1b2c3d
```

## Удаление опубликованного коммита через revert

Если коммит уже отправлен в origin и другие разработчики могли его видеть — не используйте `git reset`. Это перепишет историю, что вызовет проблемы у всех участников.

Вместо этого используйте `git revert` — он создаёт новый коммит, который отменяет изменения предыдущего:

```bash
# Отменить конкретный коммит (создаст новый коммит-отмену)
git revert a1b2c3d

# Git откроет редактор для сообщения коммита-отмены
# По умолчанию: "Revert "Исходное сообщение коммита""

# Отменить без открытия редактора
git revert a1b2c3d --no-edit

# Отмена нескольких коммитов подряд
git revert a1b2c3d..e4f5g6h
```

История при этом выглядит так:
```
C1 - C2 - C3 - C4(revert of C3)
```

Коммит `C3` не удалён — он остаётся в истории. `C4` просто отменяет его изменения. Это безопасно для публичных веток.

## Удаление коммита из середины истории (interactive rebase)

Если нужно удалить конкретный коммит не из конца истории, а где-то в середине — используйте interactive rebase:

```bash
# Запустить интерактивный rebase для последних 5 коммитов
git rebase -i HEAD~5

# Git откроет редактор:
# pick a1b2c3d Добавить авторизацию
# pick b2c3d4e Случайный тестовый коммит   ← хотим удалить
# pick c3d4e5f Исправить баг
# pick d4e5f6g Добавить логирование
# pick e5f6g7h Обновить зависимости

# Изменить 'pick' на 'drop' для удаляемого коммита:
# pick a1b2c3d Добавить авторизацию
# drop b2c3d4e Случайный тестовый коммит   ← удалить
# pick c3d4e5f Исправить баг
# ...

# Сохранить и закрыть редактор
```

Git применит все коммиты кроме помеченных `drop`. Если удаляемый коммит касался тех же строк, что и другие — могут возникнуть конфликты.

## Разница между reset и revert

Понимание различий критично для выбора правильного инструмента:

```bash
# reset — переписывает историю (опасно для опубликованных веток)
git reset --hard HEAD~1
# Результат: коммит исчезает из истории

# revert — добавляет отменяющий коммит (безопасно всегда)
git revert HEAD
# Результат: новый коммит, отменяющий изменения предыдущего
```

Визуально:

```
# До: ... - C1 - C2 - C3
# После reset: ... - C1 - C2
# После revert: ... - C1 - C2 - C3 - C3'(revert)
```

## Восстановление случайно удалённого коммита

Если вы выполнили `git reset --hard` и поняли, что удалили нужный коммит — не паникуйте. Git хранит все операции в `git reflog` в течение 30 дней.

```bash
# Посмотреть историю всех операций
git reflog

# Вывод:
# a1b2c3d HEAD@{0}: reset: moving to HEAD~2
# b2c3d4e HEAD@{1}: commit: Нужный коммит
# c3d4e5f HEAD@{2}: commit: Ещё один коммит

# Восстановить к нужному состоянию
git reset --hard b2c3d4e

# Или создать новую ветку из удалённого коммита
git checkout -b recovery b2c3d4e
```

## Практические примеры

```bash
# Просмотр перед удалением
git log --oneline -5

# Удалить последний коммит (сохранить изменения в индексе)
git reset --soft HEAD~1

# Удалить последний коммит (сохранить в файлах)
git reset HEAD~1

# Удалить последний коммит вместе с изменениями
git reset --hard HEAD~1

# Отменить опубликованный коммит
git revert a1b2c3d

# Удалить коммит из середины истории
git rebase -i HEAD~5
# Затем drop нужный коммит

# Восстановить случайно удалённый коммит
git reflog
git reset --hard b2c3d4e

# Проверить что будет удалено
git diff HEAD~1

# Удалить последние 3 коммита для новой публикации
git reset --soft HEAD~3
git commit -m "Объединённые изменения"

# Отменить revert (если revert был ошибочным)
git revert <hash-of-revert-commit>
```

## Часто задаваемые вопросы

**Какая разница между reset и revert?** `git reset` удаляет коммит из истории (переписывает историю), изменяя положение HEAD. `git revert` создаёт новый коммит, который отменяет изменения указанного коммита, сохраняя исходную историю нетронутой.

**Безопасно ли удалять опубликованные коммиты?** Удалять через `git reset` опубликованные коммиты — опасно. Это перепишет историю, что вызовет проблемы у всех, кто скачал эти коммиты. Используйте `git revert` для безопасной отмены.

**Как восстановить удалённый коммит?** Через `git reflog` — Git хранит историю всех операций. Найдите нужный хеш в reflog и выполните `git reset --hard <hash>` или создайте ветку из этого коммита.

**Что происходит с коммитами после `git reset`?** При `--soft` и `--mixed` изменения сохраняются (в индексе или рабочей директории соответственно). При `--hard` изменения из удалённых коммитов теряются. Сами коммиты остаются доступны через `git reflog` до сборки мусора (~30 дней).

**Когда использовать reset vs revert?** Простое правило: если ветка локальная и не опубликована — `reset`. Если уже в origin и другие работают с ней — только `revert`.

## Заключение

Запомните два сценария: локальный коммит — `git reset` (выбирайте `--soft`, `--mixed`, или `--hard` в зависимости от того, нужны ли вам изменения), опубликованный коммит — `git revert`. В сложных случаях `git rebase -i` с командой `drop` позволяет удалить любой коммит из истории. А `git reflog` — ваша страховка на случай ошибки.
